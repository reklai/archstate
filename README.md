# Archstate Concept

## Summary

Build archstate as a simple Go CLI for reproducible Arch package state and `.config` symlinks. V1 has no daemon or systemd service. The user manually runs `sync` to update tracked package files and `bootstrap` to reproduce the tracked setup.

Default repo location:

```text
~/.config/archstate/
```

## Repo Layout

```text
~/.config/archstate/
  .archstate-root
  pacman.conf
  aur.conf
  dotfiles.conf
  dotfiles/
    nvim/
    mimeapps.list
```

Package file mapping:

```text
package-name=description
```

Dotfile mapping:

```text
~/.config/archstate/dotfiles/<value> -> ~/.config/<key>
```

Example:

```text
# pacman.conf
git=the fast distributed version control system
neovim=vim-fork focused on extensibility and usability

# aur.conf
paru-bin=feature packed AUR helper

# dotfiles.conf
nvim=nvim
mimeapps.list=mimeapps.list
```

## Command Behavior

- `archstate init`: create `~/.config/archstate` and the default files.
- `archstate sync`: query current explicit packages and rewrite sorted package files.
  - Native explicit packages: `pacman -Qqen` -> `pacman.conf`
  - Foreign explicit packages: `pacman -Qqem` -> `aur.conf`
  - Existing descriptions are preserved by package name.
  - New descriptions are filled from package descriptions.
  - Removed packages are removed from the files.
  - Malformed package-file lines, comments, and blanks are silently cleaned up.

- `archstate bootstrap --preview`: show what bootstrap would install, symlink, replace, or prompt for without changing anything.
- `archstate bootstrap`: install missing packages and create declared `.config` symlinks.
  - Runs `sudo pacman -S --needed ...` for native packages.
  - Uses `paru -S --needed ...`, falling back to `yay`, for AUR packages.
  - If AUR packages exist but no helper is installed, stop with a clear error.
  - Package managers prompt normally; V1 has no `--yes` or confirmation-bypass flag.
  - `--adopt` and `--overwrite` are mutually exclusive dotfile conflict policies.

- `archstate dot add <name>`: manage direct children of `~/.config`; adopt existing local config into `dotfiles/<name>` and replace it with a managed symlink. If no local config and no repo target exist, report that there is nothing to add.
- `archstate dot rm <name>`: remove the entry and remove the managed symlink if present. Repo dotfile data is preserved.
- `archstate doctor`: validate repo discovery, required commands, config parseability, package access, AUR helper availability, and symlink health.

## Repo Discovery

- Commands first walk upward from the current directory until they find `.archstate-root`.
- If no marker is found, use `~/.config/archstate`.
- Later, a `--repo <path>` override can be added, but v1 should not require it.

## Dotfile Conflict Rules

- V1 supports both files and directories directly under `~/.config`.
- If `~/.config/<key>` is missing and `dotfiles/<value>` exists, create the symlink.
- If it is already the correct symlink, do nothing.
- If it exists and is unmanaged, bootstrap prompts on a TTY:
  - adopt: move the existing target into `dotfiles/<value>` when the repo target does not already exist, then symlink it.
  - overwrite: delete the existing target and replace it with the managed symlink when the repo target exists.
- Without a TTY, unmanaged dotfile conflicts fail unless `--adopt` or `--overwrite` is supplied.
- `bootstrap --preview` reports these conflicts without changing anything.

## File Rules

Every Archstate-written state file starts with:

```text
# Auto-generated by archstate.
# Treat this file as read-only state. Use archstate commands to update it.
# Manual edits may be overwritten.
```

- Files are machine-formatted and alphabetized by key.
- Existing package descriptions are preserved during `sync`.
- Comments and custom spacing are not preserved except for generated top-of-file headers.
- `sync` treats installed packages as authoritative and repairs package files.
- Commands that consume files as instructions, such as `bootstrap`, `doctor`, and dotfile commands, fail on malformed entries with file and line details.
- Writes are atomic: write temp file, fsync where practical, then rename.

## Assumptions

- V1 is Arch-only.
- No package pruning in v1.
- No systemd service or daemon in v1.
- Reproducibility means installing missing explicit packages and recreating managed `.config` symlinks.
